# 在此进行一些技术说明
## 关于模型

模型经过特殊训练，人类训练数据使用2020年以前的理工类学术论文和期刊，总共17万篇，在其中挑选了7000篇进行训练（受限于模型大小），AI训练数据使用各大主流AI平台，基于前面选择的7000篇人类论文进行改写，合并后总共1.4万篇文章，组成了原始模型的训练数据，再加上辅助算法，经测试，对于理工类学术论文的检测正确率极高，已经接近了知网和腾讯朱雀等模型，相反，对于普通文本的检测正确率有所下降，但效果仍然可观
模型训练结束后，在测试中得到了极高的成绩，loss值达到了恐怖的 '0.002283' , eval_accuracy值达到了 '0.9929' ，这意味着模型在面对 2800 篇它从来没见过的测试论文时，准确率高达 99.29%。

#### 关于辅助算法

目前AI仍然无法摆脱的一大问题就是过于自信，于是我在运算中加上了温度系数 `self.TEMPERATURE`  和指数惩罚 `self.POWER_FACTOR` 这两点措施可以在一定程度上缓解模型过于自信的情况，现在只有模型有极大的把握 >99.8% 的可能性时，才被认定为疑似AI，同时，计算文本的方差和突发性来进一步降低误判，另外，如果句中有丰富的标点符号，也会给予人类加分

## 关于自动切分

自动切分遵循 “四级智能平滑切分算法” 在遇到超长段落时，它的切分优先级如下：

第一级（按句切）： 优先找句号、问号、感叹号 。.!！?？。如果能切成正常大小，完美。

第二级（按短句切）： 如果一句话实在太长，降级去寻找逗号、分号 ,，;；，在语句的停顿处切开。

第三级（按空格切）： 如果是一串没标点的长英文，降级寻找空格，绝对保证每一个英文单词完整不被劈开。

第四级（硬切）： 只有遇到那种几千个字符完全没有空格的极端乱码（比如超长的网址），才会被迫从字符中间切断。

设计这种自动切分的功能是因为模型最多只能一次性识别512个token长度，如果一个段落超过了512token就很可能出现内存溢出，为了保证软件在任何情况下正常运作，自动切分功能是强制开启的，所以哪怕点击了合并排版按钮删除了所有换行符，软件依旧可以正常切分段落工作，目前自动切分功能只有在段落长度大于700字符时才会工作，一个中文汉字按照一个字符计数，英文字母、数字、空格、标点符号等按照0.5个字符计数
也就是说，如果一个纯汉字的段落只有汉字数量大于700字时才会触发自动切分，如果是全英文，那么这个数值将会是1400

## 关于过短忽略
同自动切分，一个中文汉字按照一个字符计数，英文字母、数字、空格、标点符号等按照0.5个字符计数，如果一个段落的字符计数 <20 ，会触发过短忽略机制，该段落的ai值不计入总ai率

## 关于总ai率

为了保证总ai率计算可靠，采用的是加权计算，总的来说就是过短的段落比重小，而长段落对结果的影响比重大，一个只有30个字的短段落如果ai率为90%，而一个有着200个字的长段落ai率为1%，那么最后总的ai率也会极低，这极大的保证了结果的准确度不受个别短段落的影响

## 关于历史记录

现在每次进行检测会保存记录，并存储到本地 `deepveri_history.json` 文件中，最多保存近10次记录，超出10次将自动删除最早的一次记录

## 关于开发者控制台

现在在界面左上角可以看见控制台按钮，每次启动模型都会先加载本地硬件，加载成功后会在左下角显示，随后可以开启控制台，控制台可以切换硬件的使用方式，切换成功后会在主界面右下角有显示，每次保存后都可以在根目录下生成或覆盖一个 `deepveri_settings.json` 文件，该文件保存了当前的设置，并且可以在下次启动时自动读取，控制台可以调整的参数说明如下：
#### 温度系数
该参数调整了模型输出结果的极端程度，换句话说就是模型的自信程度，越低越自信，最低为 `1.0` ，此数值将会使模型完全依赖自己的判断，宁可错杀也不放过一个，适合极度极端的检测场景（当然我想各位同学也用不到这个）：该数值经测试默认为 `2.0` 
#### 指数惩罚因子
该数值加大了模型过于极端的现象，但这不是坏事，这个数值只是让模型有足够的把握时才判断为ai生成，例如模型有80%的把握认为是ai，经过此参数惩罚后会进一步降低为60%甚至更低，只有模型有99.5%以上的把握时，该参数对结果的影响才会忽略不计，经测试默认为 `1.5`
#### 动态切分阈值
请阅读上方`关于自动切分` ，此数值调整的是触发切分的最大字符数，<span style="color:red">请不要调至大于`500`</span>，因为模型最多只能一次性检测512token（其实可以更多，但是为了内存不溢出在代码层面锁死了这个数值）,该数值默认为 `500`
#### 极短句屏蔽字数
这个很好理解，就是少于多少字符就不计入总的ai率，该数值默认为 `20`

### 关于开发者控制台修改默认值

为了迎合不同用户的不同模型调试需求，可以更改控制台中各个参数的默认值，并存储到本地的 `deepveri_factory_defaults.json` 中，下次再次恢复默认时将读取此文件使数值恢复到设置的值

## 关于与 TraceLess 的数据交换

这套通信机制的核心难点在于：如何让“高并发的网络请求”、“极度敏感的 GUI 事件循环”和“严苛的 GPU 张量计算上下文”这三只猛兽在一个进程里和平共处。

我们采用的是 多线程隔离 + 生产者-消费者模型 (Producer-Consumer Pattern)。

1. 线程拓扑结构 (Threading Topology)

整个软件现在在后台同时运行着三个互不干扰的线程：

>主线程 (GUI Thread)：专门负责渲染 PySide6 界面。绝对不允许执行任何耗时操作，否则界面会“未响应”。

>网络线程 (Flask Daemon Thread)：静默监听 127.0.0.1:5005。它只负责接收 HTTP 报文，将文本解析出来。

>推理线程 (Model Worker Thread)：专门负责抱着 PyTorch 模型干重活。它拥有独立的 CUDA Context（显存上下文）。

2. 通信数据流 (Data Flow) - 安全穿透机制

当 TraceLess 向端口发送一段文本时，数据经历了以下过程：

>入站 (Ingress)：Flask 线程收到 JSON，提取出 text。

>封装排队 (Queueing)：Flask 线程绝对不会直接去调用模型。它会创建一个专属的“接收篮子”(resp_q)，然后把文本和这个篮子一起打包，扔进一条全局安全传送带（queue.Queue）。随后，Flask 线程原地挂起（阻塞），安静等待。

>消费与推理 (Inference)：推理线程一直在死循环盯着传送带。看到有任务，抓过来，送进 GPU 跑前向传播（包括长文切分、特征扣除等）。

>回传 (Egress)：推理完成后，推理线程把最终的 ai_ratio 扔进那个专属的“接收篮子”里。

>出站 (Response)：挂起的 Flask 线程瞬间被唤醒，从篮子里拿出结果，包装成 JSON，通过 HTTP 200 返回给外界。

技术亮点：这种设计实现了绝对的线程安全 (Thread Safety)。由于模型始终只在自己的专属线程里工作，彻底杜绝了 PyTorch 跨线程调用导致的 Tensor 崩溃问题。

3. 状态同步机制 (Stateless to Stateful)
内存穿透：控制台修改参数时，利用了 Python 字典按引用传递 (Reference) 的特性（update 方法）。GUI 修改了参数地址里的值，后台 API 线程下一次取值时，瞬间拿到的就是新数据，实现了跨线程热更新。

4. 心跳监控 (Heartbeat)：HTTP 是无状态的，为了让主界面知道“节点连上了”，我们在 API 收到请求时，利用 PySide6 的 Signal 机制跨线程发射信号，触发主界面的“脉冲闪烁”动效，并开启 5 秒的看门狗定时器 (Watchdog Timer)，每接收到一次信号就闪烁一次。超时未收到请求，状态灯自动熄灭。
